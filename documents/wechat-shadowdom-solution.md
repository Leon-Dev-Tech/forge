# 微信浏览器 Shadow DOM 图片长按失效分析与微前端架构解决方案

## 1. 问题背景
在基于 Stencil.js 的微前端（MFE）项目中，图片组件如果嵌套在开启了 `shadow: true` (Shadow DOM) 的组件中，或者其任何祖先节点开启了 Shadow DOM，在微信浏览器中长按图片将**无法弹出保存图片/识别二维码的菜单**。

### 核心原因
微信浏览器的“长按识别”机制依赖于底层的 **Hit Testing（命中测试）**。当用户长按时，浏览器内核会尝试定位坐标点对应的 `<img>` 标签。由于 Shadow DOM 的隔离特性，微信内核无法穿透 Shadow Root 边界，导致其只能识别到自定义组件标签（如 `<my-component>`），而无法“看到”内部的图片。

---

## 2. 实验结论
通过本项目中的对比实验，我们得出以下结论：
- **Shadow DOM 阻断性**：只要图片路径上存在任何一层 Shadow DOM 祖先，原生识别即刻失效。
- **图片链接无关**：无论是 Base64 还是真实网络 URL，在 Shadow DOM 内部均无法被微信识别。
- **MFE 环境限制**：微前端沙箱通常限制使用 `window`、`document.body` 以及 `eval/Function`，导致常规的“逃离”方案失效。

---

## 3. 推荐架构方案：事件冒泡桥接 (Event Bubbling Bridge)

这是最符合微前端规范、解耦程度最高且支持无限层级嵌套的方案。

### 方案思路
子应用不再直接尝试调用微信 API，而是通过抛出一个**可穿透 Shadow 边界**的自定义事件，由拥有更高权限的“主应用（基座）”统一拦截并处理。

### 步骤 A：子应用封装通用图片组件
在 Stencil 组件中，监听点击事件并派发 `composed: true` 的事件。

```typescript
// src/components/mfe-image/mfe-image.tsx
import { Component, h, Prop, Element } from '@stencil/core';

@Component({
  tag: 'mfe-image',
  shadow: true // 开启 Shadow DOM 也不受影响
})
export class MfeImage {
  @Element() el: HTMLElement;
  @Prop() src: string;

  private handleImageClick = () => {
    // 派发自定义事件
    const event = new CustomEvent('MFE_IMAGE_PREVIEW_REQUEST', {
      detail: { current: this.src, urls: [this.src] },
      bubbles: true,   // 向上冒泡
      composed: true,  // 关键：允许事件穿透 Shadow DOM 边界
    });
    this.el.dispatchEvent(event);
  };

  render() {
    return (
      <img 
        src={this.src} 
        onClick={this.handleImageClick} 
        style={{ cursor: 'pointer' }} 
      />
    );
  }
}
```

### 步骤 B：主应用（基座）全局监听
在主应用的入口处（不受沙箱限制的环境）添加全局监听。

```javascript
// 主应用入口文件
window.addEventListener('MFE_IMAGE_PREVIEW_REQUEST', (e) => {
  const { current, urls } = e.detail;
  
  // 此时在主应用环境，可以直接访问 WeixinJSBridge
  if (window.WeixinJSBridge) {
    window.WeixinJSBridge.invoke('imagePreview', {
      current: current,
      urls: urls
    });
  } else {
    // 兜底方案：非微信环境下打开新窗口
    window.open(current, '_blank');
  }
});
```

## 4. 方案优势
1.  **架构解耦**：子应用不需要知道微信 API 的存在，也不需要 `window` 权限。
2.  **兼容性强**：完美解决 Shadow DOM 嵌套问题，微信原生预览模式支持长按、缩放和转发。
3.  **符合微前端安全规范**：不操作 `document.body`，不污染全局变量。
4.  **一次实现，多端复用**：作为通用组件库提供给其他子应用，降低整体重构成本。

---

## 5. 备选方案 (无法修改主应用时)
如果主应用无法配合修改，可采用 **`globalThis` 方案**（本项目 Demo 第 7 项）：
- 使用 `globalThis` 或 `(new Function('return this'))()` 获取全局上下文，直接在子应用内部尝试调用 `WeixinJSBridge`。
- 优点：子应用可独立上线。
- 缺点：如果沙箱极其严格（禁用 Function 构造函数），此方案可能会失效。
